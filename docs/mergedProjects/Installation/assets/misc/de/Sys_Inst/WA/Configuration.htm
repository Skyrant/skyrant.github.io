Konfiguration der Legato Webapplikation

:

Aufgrund der komplexen Parametrierung einer J2EE-Webanwendung und dem steigenden Variationsreichtum der Legato-Anwendung darf eine manuelle Konfiguration nur mit entsprechenden Kenntnissen der J2EE-Architektur und Legato erfolgen. 

Vorbereitung

Um die gelieferte War-Datei zu konfigurieren, muss sie entpackt werden.

Zum Entpacken kann sie z.B. in eine ZIP-Datei umbenannt werden.

Beim Entpacken ist darauf zu achten, dass die Ordnerstruktur erhalten bleibt.

Die entpackten Ordner sind dann in das WebApps-Verzeichnis des Applikationsservers zu speichern (tomcat*\webapps).

Konfiguration der Legato Webapplikation

Wenn Dateien manuell bearbeitet werden, ist darauf zu achten, dass deren Format nicht verändert wird.

Die Parametrierung der Webanwendung erfolgt über die Datei /WEB-INF/web.xml.

Die Funktion der Parameter ist in dem Dokument Legato_System_Parameters beschrieben.

Im Folgenden wird nur vorgestellt, wie eine DB-Verbindung und elementare Pfade eingestellt werden. Eine tiefgreifende Parametrierung erfolgt im Rahmen der Projektierungsarbeit.

Konfiguration der Datenbankverbindung

Es wird empfohlen, einen Datenbank-Pool über den Web-Application-Server zur Verfügung zu stellen. Für Legato müssen in der web.xml dafür folgende Parameter angepasst werden.

Context- Parameter DbType

Datenbanktyp MSSQL oder ORACLE

 

Folgende Parameter definieren die Verlinkung der Datenbank-Ressource auf die in Legato verwendeten Zugriffe. Der Name von res-ref-name wird mit dem entsprechenden Pool verlinkt (z.B. mit JNDI (Container)=jdbc/LegatoDB (zu suchender DB-Name)). Eine genauere Beschreibung findet sich auch im Dokument Legato_System_Parameters.

Context-Parameter DbOper

Setzen der Datenbankverbindung zur operativen DB, z.B.: JNDI=jdbc/LegatoAct 

Context-Parameter DbArch

Setzen der Datenbankverbindung zur Archiv-DB, z.B.: JNDI=jdbc/LegatoArc

für 1-DB Systeme sollte der Parameter folgendermaßen gesetzt werden:

CfgRef=DbOper (d.h.: "siehe Konfiguration von"="zu suchender Name" à in diesem Fall wird also für die Archiv-DB die Konfiguration der operativen DB verwendet)

Datenbankverbindung als kontextabhängiger Parameter

Es empfiehlt sich, die Datenbank-Verbindung über einen kontextabhängigen Parameter einzustellen. Der Vorteil ist, dass die web.xml nicht mehr jedes Mal angepasst werden muss, sondern die umgebungsabhängigen Parameter aus der Konfigurationsdatei {WEB_CONTEXT}.xml (meist Legato.xml) gezogen werden.

:

In Tomcat 6.0.30 - 6.0.32, 7.0.0-7.0.9 funktioniert dies nicht!

Wenn die Datenbank-Verbindung über einen kontext-abhängigen Parameter erfolgen soll, dann muss in der web.xml eine Referenz wie folgt angegeben werden:

 <resource-ref id=" LegatoAct_ResourceRef ">

      <description>Database connection for Legato online database</description>

      <res-ref-name>jdbc/LegatoAct</res-ref-name>

      <res-type>javax.sql.DataSource</res-type>

      <res-auth>Container</res-auth>

    </resource-ref>

<resource-ref id=" LegatoArc_ResourceRef ">

      <description>Database connection for Legato offline database</description>

      <res-ref-name>jdbc/LegatoArc</res-ref-name>

      <res-type>javax.sql.DataSource</res-type>

      <res-auth>Container</res-auth>

    </resource-ref>

TIPP:

Handelt es sich um eine 1-DB Installation empfiehlt es sich, in die context.xml folgende Parameter mit aufzunehmen, sodass keine Änderungen an der web.xml nötig sind:

<Parameter name="dboper" value="JNDI=jdbc/LegatoAct"  override="false"/>

<Parameter name="dbarch" value="JNDI=jdbc/LegatoAct"  override="false"/>

In diesem Fall ist nur die Ressource der Operativ-Datenbank zu definieren, der zweite Eintrag kann entfallen.

Sofern im Verzeichnis ${TOMCAT_HOME}\webapps\Legato\META-INF die Datei {WEB_CONTEXT}.xml vorhanden ist, kopiert Tomcat sie automatisch in das Verzeichnis ${TOMCAT_HOME}\conf\Catalina\localhost. Sofern Änderungen an der Datei vorgenommen werden, sollte die WebApp neu deployed werden, da sonst die Änderungen nicht übernommen werden.

Kontextabhängige Parameter Tomcat 8

Ab Version 8 benutzt Tomcat eine neue Version des Datenbank-Pools. Dadurch haben sich einige Parameter geändert. Ein Beispiel für die neuen Parameter sieht so aus:

Unbedingt zu beachten ist, dass der Parameter maxActive, der die Größe der Pools angibt, nun zu maxTotal umbenannt wurde. Ansonsten sind alle Pools 8 Verbindungen groß.

Grundlegende (Pfad-)Einstellungen

context-param-Parameter DefaultView

Setzen der Standardansicht (Beispiel Visu: Visu/fs/Start.svg)

 

Es empfiehlt sich, die Reports und Visualisierungen an einem anderen Ort zu speichern, um eventuelle Anpassungen daran nicht bei jedem Update der WebApp zu überschreiben. Die korrekten Verzeichnispfade müssen in den folgenden Parametern angegeben werden.

context-param-Parameter ReportOutputPath

Pfadangabe, unter der die AppEngine die generierten Berichte speichert

…\<Server>\Legato\uploads\reports

context-param-Parameter Uploadpath

Pfadangabe, unter der hochgeladene Dateien, wie Visu- und Report-Dateien gespeichert werden

…\<Server>\Legato\uploads\

Logging

Für das Logging von Meldungen aus der WebApp wird Log4j2 verwendet. Es wird eine vorkonfigurierte Datei log4j2.xml unter WEB-INF/classes mit ausgeliefert. Diese Datei kann beispielsweise so aussehen:

<?xml version="1.0" encoding="UTF-8"?>

<Configuration monitorInterval="30" status="WARN">

          <Filters>

                         <RegexFilter regex="(?is).*ORA-01013:.*" useRawMsg="false" onMatch="DENY" onMismatch="NEUTRAL"/>

<!-- Markers used in Legato web application as reference for using them to filter log messages

                         <MarkerFilter marker="exceptions"     onMatch="NEUTRAL" onMismatch="NEUTRAL" />

                         <MarkerFilter marker="init"           onMatch="NEUTRAL" onMismatch="NEUTRAL" />

                         <MarkerFilter marker="SQL"            onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="Query"          onMatch="NEUTRAL" onMismatch="NEUTRAL" />

                         <MarkerFilter marker="VisuPolling"    onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="AlarmPolling"   onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="VisuRefresh"    onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="AlarmRefresh"   onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="VisuCompiler"   onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="LegatoState"    onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="DbCache"        onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="WebServiceCall" onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="MesTermMsg"     onMatch="NEUTRAL" onMismatch="NEUTRAL" />

                         <MarkerFilter marker="MesTermErr"     onMatch="NEUTRAL" onMismatch="NEUTRAL" />

                         <MarkerFilter marker="Other"          onMatch="DENY"    onMismatch="NEUTRAL" />

                         <MarkerFilter marker="Paging"         onMatch="DENY"    onMismatch="NEUTRAL" />

-->

          </Filters>

          <Appenders>

                         <Servlet name="server">

                                        <PatternLayout charset="UTF-8">

                                                       <Pattern>[%marker] %-5level %logger{1} - %message%throwable</Pattern>

                                        </PatternLayout>

                         </Servlet>

<!-- Example for logging directly to files, instead of through the servlet engine

                         <RollingRandomAccessFile name="logfile"

                                                       fileName="webapp.log" filePattern="webapp.%i.log"

                                                       immediateFlush="false" append="true">

                                        <Policies>

                                                       <OnStartupTriggeringPolicy />

                                                       <SizeBasedTriggeringPolicy size="10 MB" />

                                        </Policies>

                                        <DefaultRolloverStrategy min="1" max="9" />

                                        <PatternLayout charset="UTF-8">

                                                       <Pattern>%date{DEFAULT} [%marker] %-5level %logger{1} - %message%n%throwable</Pattern>

                                        </PatternLayout>

                         </RollingRandomAccessFile>

                         <RollingRandomAccessFile name="errfile"

                                                       fileName="webapperr.log" filePattern="webapperr.%i.log"

                                                       immediateFlush="false" append="true">

                                        <ThresholdFilter level="WARN" onMatch="ACCEPT" onMismatch="DENY" />

                                        <Policies>

                                                       <SizeBasedTriggeringPolicy size="10 MB" />

                                        </Policies>

                                        <DefaultRolloverStrategy min="1" max="9" />

                                        <PatternLayout charset="UTF-8">

                                                       <Pattern>%date{DEFAULT} [%marker] %-5level %logger{1} - %message%n%throwable</Pattern>

                                        </PatternLayout>

                         </RollingRandomAccessFile>

 -->

          </Appenders>

          <Loggers>

                         <AsyncRoot level="trace" includeLocation="false">

                                        <AppenderRef ref="server" />

<!--

                                        <AppenderRef ref="logfile" />

                                        <AppenderRef ref="errfile" level="WARN"/>

 -->

                         </AsyncRoot>

                         <AsyncLogger name="de.gefasoft.legato.webapp.util.db.QueryWaitWatcher" level="warn"/>

          </Loggers>

</Configuration>

 

Standardmäßig wird der Appender "Servlet" benutzt. Dieser gibt die Logmeldungen weiter an die Servlet Engine, wo sie verarbeitet werden.

Unter "Filters" sollte auf jeden Fall der Eintrag "RegexFilter" beibehalten werden. Dieser filtert die Oracle-Fehlermeldungen raus, wenn der Benutzer eine Aktion abgebrochen hat (z.B. das Laden einer Seite).

Unter "Marker" sind die Logging-Kategorien aufgeführt, die vor Legato Version 4.6 in der WebApp unter dem Menüpunkt WebStatus einstellbar waren. Folgende Einstellungen sind möglich:

NEUTRAL = andere Einstellung (z.B. von "AsyncRoot") nicht überschreiben

DENY = nichts loggen

ACCEPT = immer loggen

 

Mit Log4j2 ist es nun auch möglich, die Log-Meldungen mittels "RandomAccessFile" in Dateien zu schreiben.
