Installation

Voraussetzungen

 

:

Die Versionen der AE und der Legato-Datenbank müssen gleich sein, sonst kommt es zu Fehlern.

:

Für diverse Jobs fragt die AE die Datenbank ab. Damit es hier zu keinen Problemen kommt, muss die Java-VM eine korrekte Konfiguration der Zeitzone haben. Ob überhaupt eine Zeitzonen-Einstellung vorliegt, kann über Putty mit "echo $LANG" geprüft werden. Besteht kein Zugriff auf die Installation der Java-VM, kann die Zeitzone auch im Startskript der AE (in der Datei dosvc.cmd) gesetzt werden. Dazu muss die Umgebungs-Variable JAVA_OPTS um die Angabe "-Duser.timezone=CET" ergänzt werden (ggf. ist CET durch die korrekte Zeitzone zu ersetzen).

Installation unter Windows

Unter Windows kann die Legato Application Engine mit Hilfe von Programmen der Apache Foundation (aus dem Projekt "Jakarta Commons Daemon") als Dienst installiert werden. Zur Installation gehen Sie bitte folgendermaßen vor:

Legen Sie ein Verzeichnis für die Legato Application Engine an und entpacken Sie das Archiv (LeAppEng-x.y.z.b.zip) in dieses Verzeichnis.

Kopieren Sie bitte unter Windows 32 Bit die Programme dosvc.sample.cmd, LeAppEng.exe und LeAppEngW.exe aus dem Unterverzeichnis "win", unter Windows 64 Bit die Programme dosvc.sample.cmd, prunsrv.exe und prunmgr.exe aus dem Unterverzeichnis "win64" in Ihr AE-Verzeichnis.

Speichern Sie die Dateien ojdbc*.jar und orai18n*.jar (* = aktuelle Version) in das Verzeichnis LegatoAppEng\lib. Sie können von der Oracle Homepage heruntergeladen werden. Die verwendete Version muss mindestens ojdbc6.jar (für Java 1.7) oder höher sein.

Bei einer Neu-Installation benennen Sie bitte die folgenden Dateien um:

appengine.sample.properties zu appengine.properties

log4j2.sample.xml zu log4j2.xml

Bearbeiten Sie die Datei dosvc.cmd in einem Text-Editor und nehmen Sie folgende Einstellungen vor:

:

Alle Pfadangaben müssen mit einem Backslash (\) enden.

Bei einer 1-DB Installation, löschen Sie die Angaben für "dbarch".

Bearbeiten Sie die Zeile, in welcher die Umgebungsvariable JAVA_HOME gesetzt wird so, dass JAVA_HOME auf das Verzeichnis der Java-Umgebung des betreffenden Rechners verweist. Beispiel: set JAVA_HOME= C:\Programme\Java\j2re1.7.2_08\

Falls Sie den Namen des installierten Dienstes ändern wollen, ändern Sie die Variable SVCNAME. (Falls Sie mehrere Instanzen der Application Engine am gleichen Rechner betreiben wollen, müssen Sie jeder Instanz einen eigenen Namen geben).

Falls Sie dosvc.cmd in ein anderes Verzeichnis verschieben wollen (sodass dosvc.cmd und LeAppEngBoot.jar sich nicht im gleichen Verzeichnis befinden), setzen Sie die Variable HOMEDIR auf das Verzeichnis der Application Engine.

:

Enthält der Verzeichnis-Pfad der Application Engine Leerzeichen, so führt dies zu einem Fehler. In diesem Fall muss man entweder:

das %HOMEDIR% bei SVCEXE und bei MONEXE löschen (set SVCEXE=prunsrv; set MONEXE=prunmgr)

oder man definiert im obersten Teil der dosvc.cmd zwei neue Variablen 'set SLASH=\' und 'set PLACEHOLDER="' und gibt diese immer bei Angabe des HOMEDIR mit an (z.B. set SVCEXE=%PLACEHOLDER%HOMEDIR%%SLASH%LeAppEng%PLACEHOLDER%; set MONEXE=%PLACEHOLDER%%HOMEDIR%%SLASH%LeAppEngW%PLACEHOLDER%; set SVCOPTS=--JvmOptions=-Xrs ++JvmOptions=-Dappeng.home=%PLACEHOLDER%%HOMEDIR%%PLACEHOLDER% ++JvmOptions=-Dappeng.config=%PLACEHOLDER%%CFGDIR%%PLACEHOLDER% ++JvmOptions=-Dappeng.scripts=%PLACEHOLDER%%SCRDIR%%PLACEHOLDER%)

Optional: Falls Sie die Konfigurationsdateien der Application Engine nicht im Programmverzeichnis ablegen wollen, geben Sie das Verzeichnis der Konfigurationsdateien in der Variablen CFGDIR an.

Optional: Falls Sie die JavaScript-Skripten in ein Verzeichnis außerhalb des Programmverzeichnisses legen wollen, geben sie dieses in der Variablen SCRDIR an.

Optional: Setzen Sie den Pfad für die Logdateien des Dienste-Wrapper in der Variablen LOGDIR (wirkt sich nicht auf das Logging der eigentlichen Application Engine aus; dies wird in log4j2.xml konfiguriert).

Optional: Setzen Sie die Ausführlichkeit der Logdateien des Dienste-Wrapper in der Variablen PR_LOGLEVEL (erlaubte Werte sind debug, info, warn oder error. "Debug" enthält alle Logmeldungen, also auch "info", "warn" und "error"; "info" enthält alle Logmeldungen außer "debug" usw.)

Optional: Setzen Sie Namen und Pfad der Dateien mit den umgeleiteten Standard- und Fehler-Ausgaben der Application Engine in den Variablen PR_STDOUTPUT und PR_STDERROR.

Optional: Setzen der Zeitzone. Ergänzen Sie in der Datei dosvc.cmd die Umgebungs-Variable JAVA_OPTS um die Angabe "-Duser.timezone=CET" (CET muss ggf. durch die korrekte Zeitzone ersetzt werden). Diese Angabe ist nötig, wenn die Java-VM keine oder eine falsche Konfiguration der Zeitzone hat und dies nicht in der Java-VM selbst korrigiert werden kann. Ob überhaupt eine Zeitzonen-Einstellung vorliegt, kann über Putty mit "echo $LANG" geprüft werden.

:

Wenn die Java-VM keine oder eine falsche Konfiguration der Zeitzone hat, werden keine Verfügbarkeits-Abschnitte online berechnet. Verfügbarkeitsrelevante Störungen werden dann für KPI-Berechnungen nicht berücksichtigt.

Konfigurieren Sie die Application Engine (für die genauen Schritte siehe Kapitel Konfiguration der Legato Application Engine).

Installieren Sie den Dienst, indem Sie eine Eingabeaufforderung öffnen, in das Programmverzeichnis wechseln und darin den Befehl "dosvc inst" eingeben.

Erstellen Sie eine Verknüpfung zur Monitor-Applikation im Startmenü (eventuell auch im "Autostart-Ordner" von "Alle Benutzer"). Geben Sie als "Ziel" den Befehl "dosvc monitor" (unter Windows 64 Bit "C:\Windows\System32\cmd.exe /C <Programmverzeichnis>dosvc monitor") und als Arbeitsverzeichnis das Programmverzeichnis ein.

Führen Sie die Monitor-Applikation mittels der eben erstellten Verknüpfung aus. Deren Programmsymbol erscheint daraufhin im Systembereich ("system tray") der Taskleiste. Durch Klicken mit der rechten Maustaste können Sie den Dienst starten oder dessen Konfiguration bearbeiten.

 

Der Dienst-Eintrag der Legato Application Engine wird zunächst mit der Start-Einstellung "manuell" eingetragen, das heißt, der Dienst wird nicht automatisch beim Hochfahren des Systems gestartet. Wir empfehlen, dass Sie sie erst dann auf "automatisch" stellen, wenn Sie die Application Engine vollständig konfiguriert und die Konfiguration getestet haben.

Aufruf als Applikation

Falls Sie die Application Engine als normales Programm ausführen, gehen Sie bitte folgendermaßen vor:

Legen Sie ein Verzeichnis für die Legato Application Engine an und entpacken Sie das Archiv (LeAppEng-x.y.z.b.zip) in dieses Verzeichnis (falls Sie ein JDK installiert haben, können Sie das z.B. mittels des JAR-Tools durchführen).

Kopieren Sie bitte unter Windows 32 Bit die Datei dosvc.sample.cmd aus dem Unterverzeichnis "win", unter Windows 64 Bit die Datei dosvc.sample.cmd aus dem Unterverzeichnis "win64" in Ihr Verzeichnis.

Bei einer Neu-Installation benennen Sie bitte die folgenden Dateien um:

appengine.sample.properties zu appengine.properties

log4j2.sample.xml zu log4j2.xml

 

Um die Application Engine zu starten, gehen Sie bitte wie folgt vor:

Setzen Sie die Umgebungsvariable JAVA_HOME auf das Basis-Verzeichnis Ihrer Java Laufzeitumgebung.

Rufen Sie die Application Engine auf: java -jar LeAppEngBoot.jar

 

Die Application Engine läuft nun als normale Konsolen-Applikation. Um sie zu beenden, drücken Sie Strg+C. Wenn Sie die Application Engine unter Unix oder Linux als Hintergrundprozess gestartet haben, können Sie sie durch das Signal SIGINT beenden (üblicherweise mit dem Befehl kill <process-id>). In beiden Fällen wird die Application Engine geordnet heruntergefahren, das heißt, sie beendet sich erst nach Freigabe aller Jobs.

Sie können beim Aufruf folgende Optionen angeben (vor der Angabe "-jar"):

-Dappeng.home=<Verzeichnis> - Installationsverzeichnis der Application Engine. Dies ist nötig, wenn Sie die Application Engine aus einem anderen Verzeichnis heraus starten wollen (Sie müssen dann den Pfad zu LeAppEngBoot.jar auch in der Kommandozeile angeben).

-Dappeng.config=<Verzeichnis> - Verzeichnis der Konfigurationsdateien, falls diese sich nicht im Installationsverzeichnis befinden.

-Dappeng.scripts=<Verzeichnis> - Verzeichnis der JavaScript-Skripten für die Application Engine. Nützlich, falls Sie die Skripten in einem Verzeichnis ablegen wollen, auf das auch über das Netzwerk zugegriffen werden kann und Sie nicht das Verzeichnis scripts im Programmverzeichnis freigeben wollen.

-Dappeng.properties=<Datei> - Abweichender Name der Konfigurationsdatei. Nützlich, falls Sie mehrere Instanzen der Application Engine auf dem gleichen Rechner starten wollen und dafür das gleiche Konfigurationsverzeichnis benutzen wollen (um z.B. die Datenbankverbindungen gemeinsam für mehrere Instanzen festzulegen).

-Dlog4j.configurationFile=<Datei> - Falls Sie Log4J2 für die Log-Ausgaben der Application Engine nicht über die log4j2.xml im Konfigurations- oder Installationsverzeichnis konfigurieren wollen, sondern z.B. über eine andere XML-Datei.

:

Diese Angabe muss als URL erfolgen, nicht als Dateiname (also in der Regel im Format file://<pfad>/<

datei>, mit Protokoll-Angabe file:// am Anfang und auch unter Windows mit Schrägstrich statt Backslash).

Einbinden in den Startprozess von Linux oder Unix

Um die Legato Application Engine in den Startprozess des Betriebssystems einzubinden, benötigen Sie ein Shell-Skript, das dann über symbolische Links in den entsprechenden Verzeichnissen (unter Linux üblicherweise /etc/rc.d/rc*.d) referenziert wird. Unter Linux wird die Application Engine üblicherweise in den Runlevels 3 und 5 gestartet.

Der Aufbau der Startskripten und –Verzeichnisse ist leider stark plattformabhängig. Mit der Application Engine kann folglich nur ein Beispiel-Script (leappeng.sample.sh) mitgeliefert werden, das Sie als Basis für eigene Skripten benutzen können. Die meisten Linux-Distributionen und Unix-Varianten enthalten zusätzliche Hilfsmittel, mit denen die entsprechenden Skripten stark vereinfacht werden können (z.B. startproc/checkproc/killproc in SuSE Linux, start-stop-daemon in Debian GNU/Linux, daemon/killproc in Mandrake Linux u.v.m.).

Unter Unix-Servern ist im Allgemeinen der Headless Mode von Java zu nutzen, d.h. im Aufruf der JVM sollte jeweils ein Eintrag -Djava.awt.headless=true vorhanden sein.

Im Folgenden wird als Beispiel erläutert, wie die Legato Application Engine mittels des mitgelieferten Shell-Skripts unter SuSE Linux in den Startprozess installiert wird:

Optional: Legen Sie einen Benutzer an, unter dessen Benutzerkonto die Application Engine ablaufen soll (sie soll das schließlich nicht als root tun).

Kopieren Sie leappeng.sample.sh nach /etc/rc.d/leappeng (ohne Endungen ".sample.sh") und machen Sie /etc/rc.d/leappeng ausführbar (chmod a+x /etc/rc.d/leappeng).

Bearbeiten Sie die Datei /etc/rc.d/leappeng mit einem Text-Editor. Die anzupassenden Zeilen sind im Skript als Konfigurations-Abschnitt gekennzeichnet, die anzupassenden Optionen entsprechen denen, die in den vorhergehenden Abschnitten bereits erläutert wurden.

Legen Sie eine Verknüpfung namens rcleappeng irgendwo im PATH an, z.B. /usr/sbin/rcleappeng. Testen Sie, ob Sie die Application Engine nun mittels "rcleappeng start" starten und "rcleappeng stop" beenden können. Sie sollten dazu die Application Engine bereits soweit konfiguriert haben (siehe unten), dass sie sich nicht sofort beendet. Es empfiehlt sich aber, in der appengine.properties die Instanz (AeInstance) auf eine in der Datenbank nicht vorhandene ID zu setzen, damit beim Test keine Aufträge bearbeitet werden.

Legen Sie unter /etc/rc.d/rc3.d und /etc/rc.d/rc5.d je zwei Verknüpfungen auf /etc/rc.d/leappeng an, z.B. S30leappeng für den Start und K02leappeng für das Beenden. ('S' bedeutet Start, 'K' bedeutet Kill, die Nummern geben die Reihenfolge an: ziemlich spät starten und früh beenden).

 

Bitte beachten Sie immer, dass leappeng.sh und die obigen Erläuterungen nur als Beispiel gedacht sind und keine Referenz-Implementierung für das Einbinden in den Startprozess Ihres Systems darstellen. Auch unter SuSE Linux würde man nicht unbedingt die im Skript enthaltenen Methoden für das Starten und Beenden verwenden, sondern sein Skript den dort bereits vorhandenen rc-Skripten nachempfinden, welche meist die Befehle startproc, checkproc und killproc benutzen.
