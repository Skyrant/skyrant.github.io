Task-Typen

Die Sapient Engine kann folgende Task-Typen ausführen:

calcstask: Ausführung von Berechneten Werten

dbcache: Cachen von DB-Daten

eventbus: Verwendung von Events, wie beispielsweise Meldungen oder Schichten

functions: Ausführung von Funktionen

jobs: Ausführung von Jobs

jsonapi: Ausführung von JSON-API Befehlen

jsonrpc: Ausführung von JSON-RPC Befehlen

paging: Ausführung von Funkrufen

print: Ausführung von ad-hoc-Ausdrucken

rabbitmqconnector: dient zum Verteilen von Funktionen zwischen SEs

repository: Aktualisierung des Repository

root: Basis-Task

shiftevents: Verwendung von Schicht-Ereignissen

shiftscache: Cachen von Schichtdaten

webapp: Anzeige der WebApplikation

webclient: Dient als WebClient

webserver: Dient als WebServer

 

Die Task-Typen werden mit Standard-Eigenschaften ausgeliefert. Diese werden verwendet, wenn keine besonderen Eigenschaften für die jeweilige Instanz der Task definiert wurden. Die Standard-Eigenschaften können in der Spalte "Properties" in der Tabelle se_tasks für jede Instanz individuell angepasst werden.

 

Welche Tasks jeweils ausgeführt werden sollen, kann für jede Sapient Engine Instanz separat festgelegt werden. Es kann also beispielsweise eine Instanz geben, die lediglich Jobs ausführt, ohne die Webanwendung zur Verfügung zu stellen. Diese Konfiguration erfolgt im LC2 in der Tabelle se_tasks.

Konfiguration DB-Cache

Die Task DB-Cache hat standardmäßig folgende Eigenschaften:

updateInterval = 67

updateNodeRels = true

 

Die einzelnen Werte haben folgende Bedeutung:

dbcache.updateNodeRels gibt an, ob die Tabelle L_NODE_RELATIONS synchronisiert werden soll. Ist die Synchronisation aktiv, wird der Inhalt von L_NODE_RELATIONS geprüft und repariert, falls Unterschiede zur aktuellen Knotenstruktur entdeckt werden. Dabei wird nicht die Tabelle gelöscht und komplett neu geschrieben, sondern nur differenzielle DELETE, UPDATE und INSERT Operationen ausgeführt, so dass die Daten für alle anderen Datenbank-Threads dauernd verfügbar bleiben.

:

Diese Option darf nur für genau eine Instanz der Sapient Engine aktiv sein.

dbcache.updateInterval gibt dabei das Intervall in Sekunden an.

Konfiguration Functions

Die Task Functions hat standardmäßig keine Eigenschaften. Falls gewünscht, können die folgenden Eigenschaften pro Task angegeben werden, um die verwendete Version der JavaScript-Sprache zu definieren:

js.version = 0

#js.optimization = 9

updateCheckSeconds = 0

 

Die individuellen Werte haben die folgende Bedeutung:

js.version (Integer) setzt die Version der JavaScript-Sprache, die die Sapient Engine verwenden soll. Mögliche Werte sind 0 (Rhino Default), 100 bis 180 für die JavaScript Versionen 1.0 bis 1.8, oder 200 für ECMAScript 6. Default ist 0.

js.optimization (Integer) setzt das Optimierungslevel auf Werte von -1 (Interpreter) bis 9 (optimiertes Kompilieren). Default ist 9.

updateCheckSeconds (Integer) setzt das Intervall, in dem das Repository auf Updates überprüft werden soll. Um die Prüfung zu deaktivieren, kann ein Wert kleiner / gleich 0 eingegeben werden. In produktiven Umgebungen sollte die Prüfung deaktiviert sein, da eine Überprüfung der Skripte auf Updates die Aktivität des Dateisystems erhöht.

Wegen technischer Restriktionen können weitere Einstellungen für die JavaScript Engine nur über die .properties-Datei der Sapient Engine oder durch Definition von Systemeigenschaften mit demselben Namen im Startskript getroffen werden:

js.e4x (Boolean) - XML-Verarbeitung via E4X aktivieren/deaktivieren. Default true.

js.enhancedJavaAccess (Boolean) - Zugriff auf geschützte und private Mitglieder von Java-Klassen und das Abfagen aller Java-Exceptions aktivieren/deaktivieren. Default true.

js.locationInError (Boolean) - Information über Speicherort des Quellcodes in JavaScript Fehlerobjekten aktivieren/deaktivieren. Default true.

js.parentProtoProperties (Boolean) - Die speziellen Eigenschaften _parent_ und _proto_ für JavaScript-Objekte aktivieren/deaktivieren. Default true.

js.strictMode (Boolean) - JavaScript Strict-Mode aktivieren/deaktivieren.

js.toStringSource (Boolean) - Rückgabe desselben Werts wie "toSource" für "toString" statt des normalen "[object Object]" aktivieren/deaktivieren, wenn auf Objekte oder Arrays angewendet. Default true.

js.V8Extensions (Boolean) - Zugriff auf ES6-Erweiterungen von JavaScript Engines wie V8, die noch nicht komplett ES6 unterstützen aktivieren/deaktivieren. Default true.

js.nonEcmaGetYear (Boolean) - Date.getYear() Abzug von 1900 nur für die Jahre 1900 bis 2000 aktivieren/deaktivieren, statt für alle Jahre wie von ECMA gefordert. Default false.

js.memberExprAsFunctionName (Boolean) - Die Funktion zum Festlegen von Mitgliedern als  "function object.member(..." als eine alternative Syntax zu "object.member = function(..." aktivieren/deaktivieren. Default false.

js.reservedKeywordAsIdentifier (Boolean) - Die Verwendung von reservierten Worten als Identifier aktivieren/deaktiveren. Default false.

js.threadsafeObjects (Boolean) - JavaScript Objekt-Eigenschaften-Maps als thread-sicher aktivieren/deaktivieren. Default false.

js.intWithoutDecimals (Boolean) - Rückgabe von Ganzzahlen ohne Dezimalstellen aktivieren/deaktivieren (z.B. "5" versus "5.0"). Default true.

Ist eine dieser Einstellungen sowohl in der .properties-Datei vorhanden und als Systemeigenschaft im Startskript gesetzt, dann hat die Einstellung aus dem Startskript Vorrang.

Konfiguration Jobs

Die Task Jobs hat standardmäßig folgende Eigenschaften:

schedulers = main

main.user = -1

main.startDelay = 1

main.threadCount = 10

 

Die einzelnen Werte haben folgende Bedeutung:

schedulers gibt die Scheduler an.

user gibt die Anzahl Benutzer an, die für diesen Scheduler zur Verfügung stehen.

startDelay gibt die Start-Verzögerung für diesen Scheduler an.

threadCount gibt die Anzahl der parallel ausführbaren Jobs in diesem Scheduler an. Bitte beachten Sie, dass Jobs, die mehrere Threads verwenden (SHIFT_KPI, TIMESLICES, Aggr_Poller) die Threads von dieser Task ausleihen. Daher muss die angegebene Anzahl Threads hier hoch genug gewählt werden.

 

Scheduler sind Ausführungs-Pfade in einem Programm. Auf Maschinen mit mehreren Prozessoren können Scheduler parallel auf verschiedenen Prozessoren ausgeführt werden, aber auch auf Systemen mit nur einem einzigen Prozessor haben Programme mit mehreren Schedulers Vorteile: Wenn ein Scheduler auf irgendwelche externe Ereignisse warten muss (z.B. das Bearbeiten einer Datenbank-Abfrage), kann der Prozessor in der Zwischenzeit einen anderen Scheduler ausführen.

In der Sapient Engine werden Scheduler dazu benutzt, Aufgaben (Jobs) für bestimmte Zeitpunkte zu planen und um mehrere Aufgaben parallel auszuführen. Beim Start der Sapient Engine muss mindestens ein Scheduler namens main angelegt werden, damit die Initialisierungs-Jobs überhaupt ausgeführt werden können und sich die Sapient Engine nicht sofort wieder beendet.

Der Eintrag jobs.schedulers in der Konfigurationsdatei enthält, durch Kommata getrennt, die Namen aller beim Start anzulegenden Schedulers.

Die Anzahl an Threads und Datenbankverbindungen der Task Jobs sollte größer sein als die Summe der benötigten Threads für die Jobs SHIFT_KPI, TIMESLICES, Aggr_Poller, die auf der jeweiligen Sapient Engine Instanz laufen.

Konfiguration Print

Die Task Print hat standardmäßig folgende Eigenschaften:

result.path=/web/app/assets/images/

port=4444

username=print

password=print

driver.key=webdriver.chrome.driver

driver.binary=selenium.chrome.driver-3.7.0.jar

Die einzelnen Werte haben folgende Bedeutung:

result.path gibt an, in welchem Verzeichnis die Print-Reports abgelegt werden sollen.

domain gibt die Domäne an, in der sich der Printserver befindet.

driver.binary gibt den Binär-Treiber an.

dashboard.baseurl gibt die IP-Adresse und den Port des Printservers an. Somit kann der Printserver auf einer anderen Instanz laufen. Lässt man ihn weg, wird http://localhost:8080/app/dashboards/ als Default verwendet.

Konfiguration Rabbitmqconnector

Die Task Rabbitmqconnector hat standardmäßig folgende Eigenschaften:

rabbitMqUri = amqp://rabbit:carrot@localhost:5672

exchangeName = sapient-eventbus

publishingThreadCount = 1

 

Die einzelnen Werte haben folgende Bedeutung:

rabbitMqUri gibt den RabbitMq-Server, den virtuellen Host und die Anmeldedaten an. Muss im Format amqp://user:passwort@host:port/vhost angegeben werden. Wird der defaultmäßige virtuelle Host verwendet, muss das Format amqp://user:passwort@host:port sein (ohne abschließenden Schrägstrich).

exchangeName gibt den Namen des Exchange an, der verwendet wird, um Nachrichten zu verteilen. Default ist 'sapient-eventbus'

publishingThreadCount gibt die Anzahl Threads an, die verwendet werden, um Nachrichten an RabbitMq zu senden. Default ist 1.

Der Benutzer, der verwendet wird, um sich mit RabbitMq zu verbinden, muss die folgenden Privilegien auf dem verwendeten virtuellen Host haben:

configure, write, read Berechtigungen auf den konfigurierten Exchange

configure, write, read Berechtigungen auf serverseitig erstellte Namen (beginnend mit "amq.gen") für die temporären Queues

Es empfiehlt sich, einen dedizierten virtuellen Host nur für den zugeteilten Event-Bus zu erstellen. Dann kann man dem konfigurierten RabbitMq-Benutzer configure, write und read Berechtigungen für den gesamten virtuellen Host geben, ohne dass er andere Applikationen, die denselben RabbitMq-Server benutzen, beeinträchtigen kann.

Wenn er gestartet ist, tut 'rabbitmqconnector' zwei Dinge:

Er verbindet sich mit dem Instanz-internen Event-Bus. Alle Ereignisse, die auf diesem Event-Bus veröffentlicht werden (außer denen mit aeInst=0, die nicht für andere Instanzen vorgesehen sind), werden an RabbitMq gesendet.

Er verbindet sich mit RabbitMq. Alle Ereignisse, die von RabbitMq empfangen werden (außer denen, die von der Instanz der Task stammen) werden auf dem internen Event-Bus veröffentlicht.

Nachrichten, die an RabbitMq geschickt werden, werden an einen "Fan-Out"-Exchange gesendet. Dieser Exchange kann entweder von einem Administrator erstellt werden oder automatisch durch die Task beim Start (wenn sie die nötigen Privilegien dazu hat).

Um Nachrichten von diesem Exchange zu empfangen, erstellt die Task eine kurzlebige Queue. RabbitMq entfernt diese Queue automatisch, wenn sich der Client abmeldet; es muss keine Aufräum-Aktion durch einen Administrator durchgeführt werden.

:

Der RabbitMq-Server muss entsprechend dimensioniert sein, so dass er die Menge der  von den Instanzen generierten Nachrichten bearbeiten kann. Falls der Server überlastet wird, oder falls eine Instanz zu lange die Netzwerkverbindung verliert, können Nachrichten verloren gehen.

Konfiguration Webapp

Die Task Webapp hat standardmäßig folgende Eigenschaften:

appname=Legato Sapient (engine)

x_ua_compatible=IE=edge

x_frame_options=SAMEORIGIN

content_security_policy=

content_security_policy_report_only=

# content_security_policy_report_only=default-src 'none'; script-src 'self'; connect-src 'self'; img-src 'self'; style-src 'self' 'unsafe-inline'; font-src 'self'; frame-src: 'self';

 

Die einzelnen Werte haben folgende Bedeutung:

appname gibt den Namen der Webapp an.

x_ua_compatible enthält die User Agent Kompatibilitäts-Erklärung, die in generierte Seiten eingefügt wird. Vgl. http://msdn.microsoft.com/de-de/library/cc817574.aspx). Generiert einen "X-UA-kompatiblen" Header, der die definierten Werte enthält. Es können mehrere Werte definiert und durch Semikolon getrennt eingegeben werden. Mögliche Werte sind:

 ie=edge um Microsoft Edge zu verwenden.

chrome=1 damit der IE das "Chrome Frame" Plug-in verwendet, um Seiten zu rendern.

x_frame_options steuert, ob die Web-Applikation in andere Seiten eingebettet werden darf. Da die einzelnen Ansichten ja bereits eingebettet in der Hauptseite der Web-Applikation angezeigt werden müssen, gestatten diese immer das Einbetten in Seiten der gleichen Domäne. Nur bei NONE wird auch das Einbetten in andere Seiten gestattet. Mögliche Werte sind:

DENY - Einbetten der Web-Applikation in andere Seiten nicht erlaubt.

ALLOW-FROM URL-Liste - Web-Applikation darf nur in aufgeführte Seiten eingebettet werden.

SAMEORIGIN - Einbetten nur in Seiten erlaubt, die aus der gleichen Domäne wie die Web-Applikation kommen (Default).

NONE - Einbetten der Web-Applikation in beliebige Seiten erlaubt.

Die einzelnen Werte haben folgende Bedeutung:

Die einzelnen Werte haben folgende Bedeutung:

sessiontimeout gibt an, nach wie vielen Sekunden ohne Aktivität der Webserver eine neue Anmeldung an der SE verlangt.

:

Der Pfad muss mit Vorwärts-Slashes (/) angegeben werden, Backslashes (\) funktionieren nicht!

Die Default-Einstellung maxConnection = 0 schränkt die Anzahl nicht ein. Wird dieser Wert auf eine Zahl größer 0 gesetzt, verhindert dies, dass der Webserver mehr als die angegebene Anzahl an Verbindungen gleichzeitig akzeptiert. Dies kann als Schutz gegen Dienstangriffe oder Überlastung des Servers verwendet werden.

Defaultmäßig existieren diese Einträge nicht, weil Jetty eine angemessene Heuristik verwendet, um diese Werte basierend auf den Systemfunktionen zu setzen. Um diese Einstellungen für ein Protokoll (HTTP oder HTTPS) zu ändern, müssen Werte größer 0 sowohl für "acceptors" als auch für "selectors" eingegeben werden.

auth.modules gibt die erlaubten Authentifizierungsmodule an. Eine detaillierte Beschreibung der einzelnen Optionen kann in der Installationsanleitung der Sapient Engine gefunden werden.

Sapient Engine mit TLS/SSL

Soll die Sapient Engine TLS (SSL) unterstützen, müssen ein Java Keystore und Truststore mit einem SSL-Zertifikat für den Server und den Client erstellt werden. Die Dateien sollten im Konfigurationsverzeichnis der Sapient Engine abgelegt werden. Zum genauen Vorgehen beim Erstellen des Zertifikats, siehe bitte: http://www.eclipse.org/jetty/documentation/current/configuring-ssl.html

In der Konfiguration der Tasks webserver und webclient müssen folgende Einstellungen vorgenommen werden:

Eintragen des Keystore unter dem Schlüssel "https.keystorefile", z.B. "https.keystorefile = keystore.jks".

Eintragen des beim Erstellen des Keystore angegebenen Kennworts unter dem Schlüssel "https.keystorepass", z.B. "https.keystorepass = OBF:1cj31sop1x1d1ch71cib1x191sp11cgf".

Eintragen des Truststore unter dem Schlüssel "https.truststorefile", z.B. "https.truststorefile = truststore.jks".

Eintragen des beim Erstellen des Truststore angegebenen Kennworts unter dem Schlüssel "https.truststorepass", z.B. "https.truststorepass = OBF:1cj31sop1x1d1ch71cib1x191sp11cgf".

 

Wird 'https.truststorefile' nicht angegeben, wird 'https.truststorefile' auf den Wert von 'https.keystorefile' gesetzt, 'https.truststorepass' wird auf den Wert von 'https.keystorepass' gesetzt. Dadurch wird Abwärtskompatibilität gewährleistet.

Soll ein eigenes Truststore-File verwendet werden, müssen beide Parameter angegeben werden.

Für die Task webserver müssen außerdem folgende Einträge erstellt werden:

Eintragen des TCP-Ports, auf dem die Sapient Engine TLS-Verbindungen annehmen soll unter dem Schlüssel "https", z.B. "https = 8443". Ein Wert größer 0 schaltet gleichzeitig die TLS-Unterstützung ein.

Festlegung, ob die Sapient Engine allen Verkehr auf sichere Verbindungen umleiten soll und entsprechend den Eintrag "redirsecure" auf "true" oder "false" setzen.

 

:

HTTP/2 über TLS funktioniert nur, wenn Jetty ALPN korrekt installiert wurde. Das erfordert, dass eine "ALPN boot"-Bibliothek im Class Path der JVM installiert wird (im Bibliotheksverzeichnis der SE ablegen reicht nicht). Damit nicht bei jedem Start der Sapient Engine eine Warnmeldung im Log ausgegeben wird, falls ALPN nicht korrekt konfiguriert ist, sollte in diesem Fall HTTP/2 über den Eintrag "webserver.http2 = false" in der Konfigurationsdatei ausgeschaltet werden.

 

Konfiguration Webserver Request Log

Diese Angaben betreffen die Konfiguration des Webserver Request Logs:

webserver.requestlog.enable gibt an, ob überhaupt ein solches Log geschrieben werden soll.

webserver.requestlog.extendedFmt gibt an, ob das erweiterte Format oder das NCSA Common Log Format verwendet werden soll.

webserver.requestlog.retainDays gibt an, wie viele Tage die Log-Dateien aufgehoben werden sollen.

webserver.requestlog.logCookies gibt an, ob HTTP Cookies auch geloggt werden sollen.

webserver.requestlog.timezone gibt die Zeitzone an, die für die Zeitstempel in den Log-Dateien verwendet werden soll.
