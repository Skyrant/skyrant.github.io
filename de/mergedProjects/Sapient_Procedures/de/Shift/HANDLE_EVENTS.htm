HANDLE_EVENTS (Ereignisse bearbeiten)

Beschreibung

Dieser Job dient zum Abarbeiten von Ereignissen wie Buchen von Schichtendwerten, internen Sollwerten.

Dieser Job behandelt auch die Arbarbeitung interner Sollwerte. Dazu wurde der Job erweitert, um Out-Werte beim Ereignis SHIFT_START auf übertragen zu setzen. Dadurch werden alle internen Sollwerte (ohne PROJECT_NO und COMMAND_ID) in P_VALUES_CURRENT geschrieben und aus P_OUT_VALUES gelöscht.

Sein Aufruf / Einsatz ist unabhängig vom Einsatz eines bestimmten Sapient-Moduls.

Log-Ausgaben dieses Jobs erfolgen in das Log-File der jeweiligen SE-Instanz unter dem Namen 'HANDLE_EVENTS'. Um die Ausführlichkeit der Log-Ausgaben getrennt von der anderer Jobs einzustellen, ist eine Zeile mit Inhalt "logger.de.gefasoft.sapient.db.shift.HandleEvents = <loglevel>" in die Logging-Konfiguration einzutragen.

Dieser Job kann zu einem Zeitpunkt nur einmal ausgeführt werden. Um sicherzustellen, dass es zu keinen Problemen kommt, wird der Parameter "parallel_exec" abgefragt. Liefert dieser "true" zurück, wird eine entsprechende Meldung geloggt und der Job nicht erneut angestoßen.

Beispiel:

ArgsMap args = ArgsMap.getInstance(params, DEF_SHIFT_END);

Object isParallel = args.getMapEntry("job_params", "parallel_exec");

if (isParallel != null && "true".equalsIgnoreCase( isParallel.toString() )) {

String msg = "The procedure HandleEvents/execute is executed already by another process. Please check your configuration. "

+ "Current Trigger symbol: " + String.valueOf(args.get("trigger_symbol"));

logger.error(msg);

throw new ConcurrentModificationException(msg);

}

 

Um fehlerhafte Einträge in die Datenbank am Tag vor / nach dem Produktionsdatum der Umstellung auf Sommerzeit wird als ein erster Schritt das Produktionsdatum um Mitternacht erstellt. Danach werden die Stunden und Minuten hinzugefügt. Um mehrfache Einträge derselben Schichten / Pausen ('verschobene' Schichten / Pausen am Produktionsdatum der Zeitumstellung) zu vermeiden, wird geprüft, dass dieser Eintrag noch nicht existiert.

Bedeutung der Parameter

Standardmäßig benötigt der Job keine Parameter. Es kann jedoch optional folgender Parameter eingetragen werden:

batch_size

Integer, gibt die Anzahl Zeilen an, nach denen ein Batch-Update erfolgen soll. 1 = jeder Datensatz wird ausgeführt (empfohlen für's Debuggen), Default: 1000 (empfohlen für produktive Anwendung).

Trigger

Standard-Trigger: zyklisch (-1)

Standard-Intervall: {"Interval": "1m"}
