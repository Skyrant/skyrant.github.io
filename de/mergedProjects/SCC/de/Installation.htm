Installation

Überblick

Die Installation des Cloud Connectors erfolgt über das Installationsprogramm "setupscc.exe". Eine detaillierte Installationsanleitung finden Sie im separaten Dokument "Legato_Systeminstallation".

Wird die Cloud Connector Installation zur Laufzeit (online) als Win64 Cloud Connector betrieben, sind ggf. der OPC-Server und die zugehörigen Treiber durch das Installationsprogramm Scc_opc.exe" ebenfalls am Cloud Connector PC zu installieren. Das ist nur sinnvoll, wenn der Cloud Connector im Ablauf als Windows Cloud Connector verwendet wird oder die aktuelle Installation als Projektierungsumgebung verwendet werden soll. In diesem Fall empfiehlt es sich, die Installation von Cloud Connector und OPC-Server in einem Schritt durchzuführen.

Bei Benutzung der Installation als Editor für UNIX Cloud Connector Projekte ist die Installation des Cloud Connector selbst ausreichend.

Einrichten des Cloud Connectors als Dienst

Anlegen von Diensten

In einem ersten Schritt muss eine Verbindung vom Editor zur Runtime hergestellt werden. Öffnen Sie die Datei "Scc_Edt.exe". Die Datei muss mit Administrator-Rechten ausgeführt werden. Wählen Sie unter "Datei" den Eintrag "Service Administration" aus:

Es öffnet sich folgender Dialog, in dem alle vorhandenen Dienste angezeigt werden:

Durch Klick auf "Ändern" kann ein bestehender Dienst editiert werden, Klick auf "Löschen" löscht einen Dienst.

Durch Klick auf "Hinzufügen" können neue Dienste konfiguriert werden.

Geben Sie einen Namen und eine Beschreibung für den Dienst ein. Der Service-Name muss innerhalb der Runtime eindeutig sein.

Unter Typ muss der Typ des Dienstes angegeben werden. Es werden alle gefundenen Diensttypen angezeigt. Im Folgenden sind die bekanntesten Möglichkeiten beschrieben:

Win64 Service: Standard bei Windows-Rechnern; der Cloud Connector wird als Dienst eingerichtet.

Win64 Exe: Testversion bei Windows-Rechnern; der Cloud Connector wird normal ausgeführt, mit den angegebenen Parametern. Der Cloud Connector muss manuell gestoppt werden.

Unix: Standard bei Unix-Rechnern.

LFZ Win64 Service: Dienst für den Logfilezipper.

SMC AGLink S7 Win64 Service: Maschinenclient zur Koppelung mit AGLink.

...

 

Dann wählen Sie einen Pfad aus, wohin der Dienst seine Log-Dateien schreiben soll. Standardmäßig ist hier das Verzeichnis "Eigene Dokumente" des angemeldeten Benutzers eingetragen. Darunter kann durch Klick auf ... ausgewählt werden, welche Ereignisse gelogt werden sollen.

Schließlich muss die URI der Runtime eingegeben werden (Pfad und Host).

Wenn gewünscht, kann zusätzlich ein Projekt festgelegt werden, das beim Start des Editors direkt geladen und gestartet werden soll.

Sapient Maschinenclients (SMC)

Auch Sapient Maschinenclients (SMC) können hier als Dienste konfiguriert werden. Sie dienen dazu, über Broker (z.B. Mosquitto) mit dem SCC zu kommunizieren.

Für SMCs muss zusätzlich noch eine Konfigurationsdatei selektiert werden. Durch Klick auf "Ändern..." öffnet sich ein Dialog zum Ändern der Konfigurationsdatei:

Hier muss zunächst der Name des Clients eingegeben werden. Der "Clientname" dient der Anmeldung beim Broker. Er sollte so gewählt werden, dass er auf Broker-Seite eindeutig ist.

Unter "URI" muss der Verbindungsstring zum MQTT-Broker eingegeben werden.

Außerdem kann eine Authentifizierung für den Verbindungsaufbau definiert werden, falls notwendig oder gewünscht. In diesem Fall müssen ein Benutzer und ein Passwort eingegeben werden.

Soll beim Broker eine "Last Will Nachricht" für den Fall einer Verbindungsunterbrechung hinterlegt werden, muss diese im folgenden Menüpunkt aktiviert werden. Es können dann ein Thema für die Nachricht, der Inhalt der Nachricht sowie die Qualität der Nachricht definiert werden.

Außerdem kann der Namensbereich festgelegt werden, in dem der Client mit dem Broker kommuniziert.

Schließlich müssen noch die Treibereinstellungen festgelegt werden. Es wird eine .los-Datei als Konfigurationsdatei selektiert und das Ausgabeformat des Werts festgelegt (Standardeinstellungen belassen!) Simple gibt nur den Wert zurück, Extended gibt den Wert + Buchungsdatum zurück (z.B. hilfreich, um Fehler zu finden), Complete gibt den Wert inklusive aller Details zurück.

Logfilezipper

Da die Anzahl der gespeicherten Logbücher begrenzt ist, ist der einzusehende Zeitraum bei vielen Ereignissen relativ kurz (bei großen Konfigurationen im Minutenbereich). Eine eventuell notwendige Fehleranalyse ist dann kaum möglich.

Um eine große Menge an Logdaten archivieren zu können, gibt es den Logfilezipper. Mit diesem werden die ältesten Logbücher komprimiert und in ein eigenes Verzeichnis gespeichert. Die Anzahl der komprimierten Logbücher kann parametriert werden.

Unter Unix werden entsprechende Prozesse konfiguriert und vor dem Cloud Connector gestartet.

Unter Windows können entsprechende Service Instanzen konfiguriert werden.

Der Prozess ist derselbe wie für alle anderen Dienste auch (siehe oben).

Im Bereich "Konfiguration" muss nun noch die Konfigurationsdatei für den Logfilezipper angegeben werden. Ist die Datei noch nicht vorhanden, wird sie bei Aufruf des Dialoges automatisch angelegt. Im Anschluss müssen durch Klick auf "Bearbeiten..." weitere Einstellungen vorgenommen werden:

Unter "Verzeichnis Logdateien" wird das Verzeichnis gewählt, das überwacht werden soll. Dieses Verzeichnis muss bereits vorhanden sein!

Unter "Verzeichnis Zipdateien" wird das Verzeichnis angegeben, in dem die komprimierten Dateien abgelegt werden sollen. Dieses Verzeichnis muss noch nicht existieren, es wird dann beim Start angelegt.

In "Dateiname" muss der Name der Datei angegeben werden, welche bei Änderung komprimiert wird. Diese Datei ist im Regelfall die letzte Logdatei (bei Meldungslogs name+".log.019", bei Fehlerlogs name+"err.09").

Die "Max. Anzahl Zipdateien" gibt an, wieviele Dateien im Zip-Verzeichnis vorgehalten werden.

Schließlich muss noch der Pfad zum Zip-Programm angegeben werden. Derzeit kann nur gzip.exe verwendet werden bzw. ein Programm, welches die gleichen Aufrufparameter verwendet. gzip.exe wird mit der SCC Runtime mit ausgeliefert.

Dienst installieren

Der Dienst ist nun konfiguriert, aber noch nicht installiert. Selektieren Sie den Dienst im mittleren Teil des Dialogs und klicken Sie dann unten auf "Installieren".

Der Dienst wird installiert; der Text unten ändert sich von "Dienst nicht installiert" zu "Dienst gestoppt".

Dienst starten

Klicken Sie nun auf "Start", um den Dienst zu starten.

Der Text ändert sich zu "Dienst läuft". Er kann nun bei Bedarf gestoppt oder pausiert werden. Dies kann auch über das Dienste-Menü von Windows erfolgen.

Beenden Sie den Dialog mit "OK".

Verbinden zur Runtime

Nun muss sich der Editor mit der gewünschten Runtime verbinden. Öffnen Sie dazu das Menü "Datei" - "Mit Service verbinden ..."

Unter Service-Name kann nun aus einer Drop-Down-Liste der gewünschte Dienst ausgewählt werden (es stehen alle Dienste zur Verfügung, die zuvor eingerichtet wurden).

Darunter muss noch die URI des Dienstes eingegeben werden (Pfad zur Runtime und Port des Redis).

Der Editor ist nun mit der Runtime verbunden. Nun kann ein Projekt geladen und gestartet werden.
